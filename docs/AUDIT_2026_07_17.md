# ParcelLogic Full Audit — 2026-07-17

**Scope:** end-to-end audit of ParcelLogic (sourcing) + Stack Portfolio Dashboard (pipeline), commissioned by Nache with three questions: (1) why is the app strong for self-storage but weak for Luxury Garage Condo (LGC)? (2) how do we make it more automated and effective? (3) will it get us to 115 facilities by 2040 (currently ~20 incl. pipeline)?

**Headline:** the LGC weakness is not a data gap — it's a plumbing gap with one genuine data defect. The zoning matrix already stores an LGC verdict for every grounded municipality, derived by almost exactly Nache's own business rule. But every downstream surface (scoring, needles, Hot Deals, alerts, dashboard push) reads only the `self_storage` column, and the anti-poison gate has been quietly forcing the stored LGC verdict to `prohibited` in exactly the zones that matter most (light-industrial-viable, storage-dead). The fix requires **zero re-grounding** of the 58 counties.

---

## 1. What's strong (keep and protect)

- **Grounding discipline.** Human-reviewed verdicts with verbatim citations, hard anti-poison gates (`postingest_gate.py`), one-command batch verification (`verify_batch.py`), casing checks, closed-list sweeps, the catch ledger. This is the moat — a plausible-but-wrong verdict is the one failure the gates can't catch, and the process is built around preventing it.
- **The wealth-ring needle metric.** One number (grounded storage-viable + ≥1.5 ac + dt=10 ring HV ≥ $475k / HHI ≥ $100k) that maps directly to "parcels worth handing to outreach." The ring precompute (`parcel_ring_metrics`) is asset-agnostic and reusable as-is for LGC.
- **CoStar integration.** Upload → six-tier matcher → owner/broker-phone/last-sale enrichment (shipped 2026-07-16) → on-needle deal pool. 38 counties swept; ~120 on-needle contactable deals.
- **Deal Pipeline bridge.** Buy-box deals push into the dashboard's `deal_prospect` board with a clean facts-vs-disposition contract, verdict trust badges, owner gating, and one-click branded LOI generation.
- **Ingest pipeline (Track 1).** New-jurisdiction ingest (ArcGIS discovery → parcels → spatial backfill → LLM ordinance parse → overlays → ring precompute → auto-score) is genuinely one API call.

## 2. LGC diagnosis — written everywhere, read nowhere, suppressed where it counts

### 2.1 The stored verdict exists and mostly encodes the right rule

`zone_use_matrix.luxury_garage_condo` is populated on every muni apply. The parser inference (`app/services/ordinance_parser.py:282-309`) is nearly verbatim Nache's rule ("if it's zoned warehouse we claim warehouse; LI we spin as LI; storage-by-right we claim a storage-type use"):

| Ordinance says | Stored LGC verdict |
|---|---|
| mini_warehouse or self_storage permitted | permitted |
| mini_warehouse or self_storage conditional | conditional |
| light_industrial permitted/conditional | conditional |
| none of the above | prohibited |

### 2.2 But nothing downstream reads it

| Pipeline stage | What it reads | LGC status |
|---|---|---|
| Buy-box scoring (`buybox_scoring.py:300,324-333`) | `zum.self_storage` literally | absent; `UseCase.use_keys` designed for this, ignored |
| Use cases (migration `0015:193-200`) | only `self_storage` ever seeded | no LGC use case exists |
| Needle metric (`verify_batch.py:44-63`) | `v.ss IN (permitted,conditional)` | never tallied |
| Hot Deals digest (`daily_email.py:369-378,415-433`) | self_storage LATERAL + `storageVerdictMode` | no LGC lane |
| Listing alerts (`listing_alerts.py:47`) | self_storage-derived score ≥ 85 | blind |
| Dashboard push (`dashboard_push.py:50-57`) | digest's selection; no `asset_type` column | board can't tell LGC from storage |
| CoStar `property_type`/`secondary_type` (`forsale_listing.py:65-66`) | stored, never consumed | unused signal |
| Frontend (`dashboard/[jobId]/page.tsx` ~298) | `target_use` **hardcoded** `"self_storage"` | matrix column display-only |
| Product docs (`docs/TARGET_MARKETS.md`) | storage-needle framing | LGC absent as a target |

### 2.3 And the gate suppresses the most valuable slice (the one real data defect)

Catch #58 (`postingest_gate.py:57-70`) hard-fails any stored LGC verdict of permitted/conditional when **both** storage siblings are prohibited, unless the ordinance names a garage use. That is the correct rule for stored-verdict integrity (it exists to stop inference leaks) — but it contradicts the parser inference in one shape: **light-industrial-viable, storage-dead zones**. To pass the gate, human applies forced `lgc=prohibited` there (e.g. `_apply_cary_nc.py`: "lgc prohibited throughout"). Those zones are precisely the incremental LGC pool: land where storage can't go but a garage-condo entitlement argument can be made.

**Consequence: the stored `luxury_garage_condo` column cannot be trusted for LGC sourcing.** LGC-viability must be derived at query time from the three sibling columns (`self_storage`, `mini_warehouse`, `light_industrial`) — which are trustworthy, human-reviewed, and gate-protected.

## 3. Automation friction (per-muni grounding is the cost center)

1. **Grounding is a script farm.** ~200 hand-written `_apply_<muni>.py` files (of 482 in `backend/scripts/`); every deal-producing muni costs a human adjudication plus a bespoke Python file. The op5 factory orchestrator + `/admin/op5-review` bulk-approve are the seed of the fix but don't cover fetch/adjudicate, and a data-driven verdict applier (verdicts JSON → rows) doesn't exist yet.
2. **Ordinance fetch is manual for JS sites.** `ordinance_fetcher.py` is httpx+BeautifulSoup; the Municode `CodesContent` API, eCode360 print-endpoint, and Playwright-defeats-Cloudflare unlocks live as operator curl recipes in CLAUDE.md instead of code.
3. **The digest cron rides a watchdog.** `daily_email.py` fires from `queued_job_watchdog.py`'s tick because Railway's cron startCommand never took effect (4-day dark stretch in May). Works, but fragile.
4. **Census geocoder caps CoStar match rate.** Serialized 1 req/s, in-process cache lost on every restart, silent `None` on transient errors, no fallback — while Mapbox is already a dependency. Parcels ingested without addresses amplify it (geocode-only matches cap at 0.75 confidence, under the 0.85 gate).
5. **Listing-match worker bypassed by hand.** The Dramatiq actor dies on county-sized geocode tiers (catch #25), so CoStar ingest is upload + hand-run `_match_listings_direct.py`.
6. **Watchdog detects, doesn't remediate.** Stuck jobs still need manual `cancel_stuck_jobs.py` / `recover_stale_jobs.py`.

## 4. Dashboard gaps (the loop doesn't close)

- **One-way sync.** ParcelLogic writes `deal_prospect` directly into the dashboard's Supabase; dispositions (passed, under-contract, LOI-sent, notes) never flow back, so ParcelLogic keeps re-surfacing dead deals and can't learn conversion rates.
- **No asset type on the board.** `deal_prospect` has no way to distinguish a storage deal from an LGC deal.
- **No push health.** No last-push timestamp, no error record, no delisted-parcel cleanup (rows are only ever upserted).
- **The 115 goal is half-wired.** "Delivered" is live, but the in-flight pipeline is a hardcoded TypeScript array (`frontend/lib/goals.ts:33-47`), and `deal_prospect` doesn't feed the goal math at all.
- **No needles-by-county view anywhere.** The product's one metric lives in `verify_batch.py` stdout and markdown files.

## 5. Funnel math to 115

95 builds to go in ~14 years ≈ **6.8/year sustained** (currently ~2-3/yr implied by ~20 since inception). At today's numbers there are **207 on-needle contactable deals on the market** (122 storage + 85 LGC-only) plus **16,816 wealth-gated needles** available for off-market outreach. If outreach → LOI → contract → close converts at even 5-10% end-to-end, the *listed* pool alone supports 10-20 closings/year — comfortably above the 6.8/yr requirement — but only if (a) the LGC lane opens (opening it grows the listed pool 70% and the needle pool 57% overnight, no re-grounding), (b) off-market outreach starts systematically working the armed needles (owner mail/call campaigns keyed off the CoStar owner/phone enrichment), and (c) dead deals stop recycling through the board (two-way sync). The binding constraint is **not** parcel supply — it's lane visibility (LGC) and outreach conversion. Instrument the funnel (Phase 5) and revisit against actuals.

## 6. Roadmap (priority order)

### Phase 1 — LGC parity (ships first; zero re-grounding, zero risk to the storage path)

Core mechanism: **query-time derived verdict** via a new slug-keyed registry (`backend/app/services/use_verdicts.py`) — `self_storage` renders byte-identical SQL to today; `luxury_garage_condo` renders the sibling-column CASE (Nache's rule verbatim, immune to catch-#58 suppression). Stored `lgc` column and catch #58 stay untouched (stored lgc becomes advisory/display).

| Step | What ships | Effort |
|---|---|---|
| 1a | **In-app asset toggle** — `luxury_garage_condo` in `TargetUse` + derived `case()` in `candidate_search.py`; toggle in the dashboard header re-slices map + table. Owner sees every LGC-viable parcel, including the hidden LI-only pool, immediately. No migration. | ~1 day |
| 1b | **LGC scores** — migration `0047` seeds the LGC use case + "LGC Default Box" default filter; `buybox_scoring.py` resolves filter→use-case slug and templates the verdict expression; `auto_score_jurisdiction` loops all default filters; backfill via existing `_score-all` endpoint. `parcel_buybox_scores` is filter-keyed, so zero collision. Regression test: rendered self_storage SQL byte-identical. | 1-2 days |
| 1c | **LGC digest + alerts, no double-email** — new `excludeStorageViable` filter key restricts the LGC lane to the incremental pool (LGC-viable AND storage-dead); both-viable parcels keep arriving once via the storage digest. Migration `0048` seeds "LGC Hot Deals" with email OFF until smoke-tested. | 1 day |
| 1d | **Asset tag on the board** — `asset_type` (`self_storage`/`luxury_garage_condo`/`both`) as a fact column; single row per parcel (disposition is per-parcel). **Deploy dashboard-repo migration first**, then the push change. | 0.5 day |
| 1e | **LGC needle tally** in `verify_batch.py` (total + incremental per town), additive. | 0.5 day |

### Phase 2 — Deal flow & outreach
Two-way dashboard sync (dispositions back to ParcelLogic; passed deals stop resurfacing; conversion telemetry), LOI-sent status automation, delisted-parcel cleanup + push-health record, owner-contact enrichment beyond CoStar (skip-trace integration), off-market outreach queues from armed needles.

### Phase 3 — Onboarding automation
Codify the fetch unlocks (Municode `CodesContent`, eCode360 print endpoint, Playwright for Cloudflare-JS) into `ordinance_fetcher.py`; data-driven verdict applier (verdicts JSON + citations → matrix rows) to retire per-muni `_apply_*.py`; make the op5 factory the default county path with the review queue as the only human step.

### Phase 4 — Freshness & reliability
Mapbox/Google geocoder fallback + persistent geocode cache (direct CoStar match-rate lift); fix the listing-match worker (or bless `_match_listings_direct.py` as a queued job); real scheduler for the digest; watchdog auto-remediation; periodic ordinance-amendment re-checks for grounded munis.

### Phase 5 — Goal tracking to 115
Needles/deals-by-county view in-app (replace CLI-only reporting); wire `deal_prospect` + real pipeline statuses into the dashboard's goal math (retire the hardcoded `PIPELINE` array); funnel-conversion instrumentation (needle → contacted → LOI → contract → close) so the 6.8/yr requirement is tracked against actuals.
