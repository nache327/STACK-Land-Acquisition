### Viability verdict

**YES.** The PDF source is viable, and the right primitive is better than Lane A's original "manual digitization" wording:

1. Extract vector geometries from `Layers_Zoning_Designation1`.
2. Preserve `OGR_STYLE`.
3. Classify polygons by fill/hatch style and map styles to the legend zone codes.
4. Dissolve polygons by `zone_code`.
5. QA against the rendered PDF, MetroGIS parcels, city boundary, streets, lakeshore, and a parcel-centroid overlay.

This is not a no-QA automation path because the PDF does not expose a semantic `zone_code` attribute. The risk is style-to-code assignment, not boundary tracing.

## Recommended authoring workflow

### Path A - GeoPDF vector extraction, preferred

1. Download the official March 2025 zoning map PDF from Wayzata.
2. Use GDAL/OGR to extract `Layers_Zoning_Designation1` to GeoPackage or GeoJSON while preserving `OGR_STYLE`.
3. Filter to polygon and multipolygon features; discard label strokes, roads, legend boxes, and line-only symbology.
4. Build a style-to-zone-code crosswalk from the rendered legend. Expect manual review for hatches because some hatches appear as separate line features.
5. Assign `zone_code`, dissolve by `zone_code`, and clip to the Wayzata municipal boundary layer exposed in the same PDF or to the registered Wayzata jurisdiction bbox.
6. Run Lane A's Class A pre-flight before any production backfill:
   - district bbox covers >=50% of parcel bbox
   - all-Wayzata or 1,000-parcel `ST_Within` dry run >=50% match
   - check contained coverage expectation against the 70% operational gate
7. Spot-check 20-30 parcels across lakefront, downtown, and western residential areas against the rendered PDF.
8. Ingest districts with provenance:
   - `source_url`: March 2025 Wayzata zoning-map PDF
   - `source_vintage`: 2025-03-25
   - `source_method`: GeoPDF vector extraction + style-to-zone-code QA

Estimated effort: **2-4h** if style mapping is straightforward; add **1-2h QA** if hatches/overlays are ambiguous. This should still beat hand-tracing.

### Path B - QGIS manual cleanup / digitization fallback

Use this only if the OGR vector export contains too many cartographic artifacts to clean efficiently.

1. Load the GeoPDF directly in QGIS; it should already land in Hennepin feet.
2. Overlay MetroGIS/Hennepin parcels and Wayzata city boundary.
3. Trace/dissolve district boundaries by zone-code family, using streets, parcels, shoreline, and city boundary as snapping references.
4. Export one polygon layer with `zone_code`.
5. Run the same pre-flight and parcel spot-checks as Path A.

Estimated effort: **4-6h** for production-quality output, not just 3-4h, because QA and style/code validation matter more than raw tracing speed.

## Lower-cost partial fallback

Rejected for customer-facing operation.

Wayzata's 1,976-1,992 parcels likely have dominant residential zones plus downtown commercial, parks, institutional, and PUD pockets. A broad-stroke fallback could label large residential areas using the map image, assessor use classes, and neighborhood location, then leave hard downtown/PUD/lakeshore parcels residual.

Why this is not recommended:

- It would infer parcel zoning from location and property use rather than authoritative zoning geometry.
- It would create a truthfulness problem if used to clear the operational gate.
- It would be harder to explain than the GeoPDF extraction path, which is both available and official.

Acceptable limited use: internal triage only, or a clearly labeled residual-analysis artifact if Master chooses to defer production Wayzata while keeping an estimate of likely coverage.

## Bainbridge precedent check

Bainbridge Island did not use a city PDF or a reusable vendor-published service. It used Washington's WAZA statewide FeatureServer:

```text
https://services6.arcgis.com/tboeqGwETr5ppr5Q/arcgis/rest/services/WAZA_Prototype_Layers/FeatureServer/0
filter: Jurisdiction='Bainbridge Island'
zone field: ZoneID
```

That source is Washington-specific and does not transfer to Wayzata. The closest Minnesota analog checked in prior Hennepin work was MetroGIS, but MetroGIS is parcel-only and carries assessor-use fields, not zoning districts.

Bainbridge lesson that does transfer: do not assume city ordinance codes and aggregator codes match. For Wayzata, there is no aggregator-code mismatch because the PDF legend and Municode Chapter 937 are the source pair. The mismatch risk is instead **style-to-zone-code assignment** from the GeoPDF.

Proceed with Wayzata instead of deferring it.
